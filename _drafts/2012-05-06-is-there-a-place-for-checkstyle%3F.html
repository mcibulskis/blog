--- layout: post title: Is There a Place For Checkstyle? date: '2012-05-06T11:14:00.002-07:00' author: Michael Cibulskis tags: - Checkstyle - Agile - Craftsmanship - Metrics modified_time: '2012-05-24T21:04:30.200-07:00' blogger_id: tag:blogger.com,1999:blog-4949339273493077325.post-5950004686545105913 --- with modern ides, is there really a place for checkstyle anymore/things are easily reformatted by a simple command within ides, so is there really any reason that things like extraneous spaces, spaces before or after symbols, etc should be flagged as checkstyle errors/  one could argue that terribly offensive code should be flagged and never checked into the code repository, and i would agree with that.  however, it should be their pair partner who should be preventing them from doing that, or failing that, their other team members.  any kind of major readibility issues with code should be a team or pairing patner action.  if the team cannot or is not willing to take responsibility for the readability of their code, then the team has much larger issues, which will surface as code maintainability problems, loss of velocity as stories revisit previously written code to enhance functionality, and likelihood of errors in test cases and production code.
